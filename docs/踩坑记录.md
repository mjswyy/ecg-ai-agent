## 0. 开发阶段踩坑（Phase 1-2 代码开发与调试）

### 0.1 wfdb 库自动去除 .hea 文件中的 `#` 前缀

**日期**：2026-07-07

**现象**：`ECGLoader._parse_metadata()` 始终无法解析 Age/Sex/Dx 等字段，所有 record 元数据均为空。

**原因**：`.hea` 文件的注释行格式是 `# Age: 74`，但 `wfdb.rdrecord()` 库在读取 comments 时**会自动去掉 `# ` 前缀**，返回的 comments 格式为 `Age: 74`。代码最初用正则 `rf"#\s*{field}\s*:\s*(.*)"` 匹配，自然失败。

**解决**：将正则改为 `rf"^{field}\s*:\s*(.*)"` 匹配 wfdb 实际返回的格式。

**教训**：第三方库的 API 文档与实际返回数据格式可能不完全一致，首次集成时需要手动检查返回值的具体内容。

### 0.2 `.mat` 文件不是 MATLAB 格式

**日期**：2026-07-07

**现象**：用 `scipy.io.loadmat()` 读取 PhysioNet 的 `.mat` 文件直接报错。

**原因**：PhysioNet 的 `.mat` 文件是 **WFDB format 16** 二进制格式（16 位有符号整数、一阶差分编码、12 通道交错），与 MATLAB 的 `.mat` 格式完全不同，仅仅是扩展名恰好也叫 `.mat`。

**解决**：必须使用 `wfdb.rdrecord()` 读取，该库内部自动完成差分解码和导联分离。跳过 `.mat` 扩展名，直接传 `.hea` 路径的无扩展名部分给 wfdb。

---

### 0.3 Windows 上简繁体汉字创建出两个不同的目录

**日期**：2026-07-07

**现象**：项目目录结构有两份，部分文件在 `ecg资料/` 下，部分在 `ecg資料/` 下，看似同一路径实则两个独立文件夹。

**原因**：Windows 文件系统中 `资`（U+8D44，简体）和 `資`（U+8CC7，繁体）是两个不同的 Unicode 字符，可以共存。在编辑器和终端中肉眼几乎无法区分。写文件时工具链不一致导致了路径分叉。

**解决**：用 `find` 列出两份目录的全部文件，用 `cp -r` 合并到简体目录，`rm -rf` 删除繁体目录。

**教训**：Windows 上避免文件路径包含 CJK 字符，或使用纯 ASCII 目录名。

### 0.4 SNOMED CT 代码 `251146004` 被重复映射到两个不同疾病

**日期**：2026-07-07

**现象**：`LabelExtractor` 只加载了 26 个类别（应为 27），且第 27 类 "Incomplete Right Bundle Branch Block" 消失。

**原因**：`CHALLENGE_CLASSES` 字典中，SNOMED CT 代码 `251146004` 同时出现在 "Incomplete Right Bundle Branch Block"（传导类）和 "Left Ventricular Hypertrophy"（形态类）两个条目下。Python 字典重复 key 时后者覆盖前者，IRBBB 被 LVH 覆盖后消失。

**解决**：对照 PhysioNet 2020 Challenge 官方评分代码确认：`251146004` 的正确映射是 IRBBB。删除 LVH 条目（LVH 不在 Challenge 27 类中），并新增缺失的第 27 类 `164884008`（Ventricular Ectopic Beats，VEB）。

### 0.5 零填充后的归一化失真（信号幅度放大 ~2 倍）

**日期**：2026-07-07

**现象**：预处理后短信号的归一化结果异常，std 远小于 1.0。

**原因**：预处理中短于 4096 样本的信号会被对称零填充。代码最初对**整个填充后的数组**（含大量 0 值）计算 mean/std。对于 1000 样本的信号+3096 个零填充，std 被大约稀释 50%，导致 z-score 归一化后真实信号区域被放大约 2 倍。

**解决**：修改 `_normalize()` 方法，接收 `pad_left`/`pad_right` 参数，**仅对非填充的信号区域计算统计量**，再将归一化公式应用于整个数组。

**教训**：信号处理中零填充区域的统计量会严重扭曲归一化结果。任何需要全局统计量的步骤必须注意填充区域的影响。

---

### 0.6 Age 字段的 NaN 值毒化下游统计

**日期**：2026-07-07

**现象**：数据集统计中 `age_stats.mean` 为 NaN，`get_statistics()` 返回异常。

**原因**：ECGSample.age 属性中 `float("NaN")` 返回 Python `nan`（IEEE 754），该值 `is not None` 的检查通过，被当作有效年龄进入 `ages` 列表。`np.mean([..., nan, ...]) = nan`。

**解决**：在 age 属性中添加 `np.isnan(val) or np.isinf(val)` 检查，NaN/inf/负数/>122 岁统一返回 `None`。同时在下游统计代码中过滤 `None` 值。

### 0.7 SNOMED CT 等价映射缺失导致 75% 代码未覆盖

**日期**：2026-07-07

**现象**：初次全量预处理后，~9% 的训练记录标签全为 0（零标签率过高）。

**原因**：PhysioNet 数据集包含 107 种不同的 SNOMED CT 代码，但初始 `CHALLENGE_CLASSES` 只定义了 26 个主代码的一对一映射。大量等价/变体代码（如 `39732003` → LBBB、`428750005` → VEB）未被映射，且错误地将一些规范代码映射为等价代码（如将 Sinus Bradycardia 映射到 Sinus Rhythm，几乎导致 6 个类别被合并）。

**解决**：新增 `SNOMED_CT_EQUIVALENTS` 等价映射表（15 个映射），重构 `_full_mapping` 支持两阶段查找（规范→完整）。`save_mapping()`/`load_mapping()` 同步持久化等价映射。最终零标签率从 9% 降到 0.4%。

### 0.8 ASL Loss 中 `torch.log(0)` 产生 NaN

**日期**：2026-07-11

**现象**：训练评估时 `roc_auc_score` 抛出 `ValueError: Input contains NaN`。

**原因**：AsymmetricLoss 中概率偏移计算 `1 - p + self.clip`，当模型预测概率 p 接近 1.0 时，该值可能 ≤ 0。代码只 clamp 了 `max`（上限），未 clamp `min`（下限），导致 `torch.log(0 or negative) = -inf or NaN`。

**解决**：将 `torch.clamp(1 - p + self.clip, max=1.0-eps)` 改为 `torch.clamp(1 - p + self.clip, min=eps, max=1.0-eps)`，同时 clamp 上下限。

---

### 0.9 CUDA 13.0 + AMP 混合精度训练导致模型全 NaN

**日期**：2026-07-12 ~ 2026-07-15

**现象**：A100（CUDA 13.0）上训练 InceptionTime：

- 每 epoch 评估时 **全部 174,474 个 logits 都是 NaN**（验证集 6,462×27 类）
- val_f1 永远卡在 0.1181（随机基线）
- val_auc = 0.5000（完全随机）
- 训练 loss 快速降到 0.0000（过拟合训练集 + NaN 污染）

**原因**：A100 预装 CUDA 13.0 是非常新的版本，PyTorch 的 `torch.cuda.amp.autocast` 在该版本下存在兼容性问题。AMP 在 forward pass 中将部分 FP16 操作降级到异常精度，导致梯度下溢为 NaN。虽然加了 NaN 保护跳过坏 batch，但之前的 NaN 已经通过梯度更新污染了模型权重。

**解决**：在训练命令中添加 `--no-amp` 参数，关闭混合精度，使用纯 FP32 训练。InceptionTime 仅 291K 参数，不需要 AMP 加速。

> **2026-07-15 更新**：Ascend NPU 上出现了完全相同的症状（174K NaN logits + val_auc=0.5000），但根因不同 —— 是预处理数据中 8 个 .npy 文件含 NaN，而非 AMP。数据 NaN 修复后 AMP 在 NPU 上正常可用（2 epoch 无 NaN，val_auc=0.585）。详见 [1.6](#16-预处理数据含-nan-导致训练全量静默失败)。**相同症状不等于相同根因**。

### 0.10 正则预编译与 `get_record_count` 内存优化（低优先级）

**日期**：2026-07-07

**现象**：预处理脚本 CPU 时间长，`get_record_count` 消耗过大的临时内存。

**原因**：
- `_parse_metadata` 每次调用都重新 `re.search` 编译正则（6×N 次编译，N 为 comment 数），43K 条记录累积开销可观
- `get_record_count` 用 `len(list(rglob(...)))` 将全部 Path 对象 materialize 为列表再取长度

**解决**：在 `__init__` 中预编译正则模式列表 `_meta_patterns` 复用；`get_record_count` 改用 `sum(1 for _ in rglob(...))` 懒计数。

### 0.11 `set()` 迭代顺序不确定导致数据划分不可复现

**日期**：2026-07-07

**现象**：运行两次 `preprocess_data.py`，相同随机种子下 train/val/test 划分却不同。

**原因**：`sources = list(set(m["source"] for m in manifest))` 中 `set()` 的迭代顺序依赖 Python 进程的 hash seed，每次运行不同。即使 `RandomState(42)` 固定，按不同顺序遍历数据源导致的划分也不同。

**解决**：改为 `sources = sorted(set(...))` 固定顺序。

**日期**：2026-07-07

**现象**：预处理 43,101 条记录全部完成后，保存标签映射时崩溃，所有已处理数据丢失。

### 0.13 NaN/Inf 年龄导致 JSON 序列化失败

### 0.14 `except Exception` 吞掉 Ctrl+C

**日期**：2026-07-07

**现象**：预处理中途按 Ctrl+C 无法终止，脚本继续处理下一条记录。

**原因**：`preprocess_data.py` 中的 `except Exception as e` 会捕获 `KeyboardInterrupt`（Ctrl+C 引发的异常），将其当作普通错误记录为 "skipped" 并继续。

**解决**：添加 `except (KeyboardInterrupt, SystemExit): raise` 在上层重新抛出中断信号。

### 0.15 `resample_poly` 输出长度与预计算 `new_len` 不一致

**日期**：2026-07-11

**现象**：升级重采样算法从线性插值到 `scipy.signal.resample_poly` 后，预处理部分记录崩溃 `ValueError: could not broadcast input array`。

**原因**：`resample_poly(up, down)` 输出的精确长度是 `ceil(orig_len * up / down)`，而代码预计算的 `new_len = round(orig_len * target_fs / orig_fs)` 是取整值。两者因浮点误差偶尔相差 1 样本。

**解决**：不再预计算精确长度，改为让 `resample_poly` 自然输出后 `[:target_len]` 裁剪或零填充对齐。

### 0.17 USTC 算力平台（107）特有的 SLURM 配置

**日期**：2026-07-12

**现象**：
- `sbatch: error: QOSMaxMemoryPerUser` — 内存请求超过学生分区限制
- `sbatch: error: QOSMaxCpuPerUserLimit` — CPU 请求超过限制
- `sbatch: error: Job violates accounting/QOS policy` — partition 名称不匹配

**原因**：USTC 107 平台的学生分区参数与通用 SLURM 脚本不同：
- partition 应为 `Students`（非 `gpu`）
- 需额外指定 `--qos=qos_stu_default`
- 内存上限 16G（非 32G）
- CPU 核数上限 4（非 8）

**解决**：修正 SLURM 脚本头部分区配置，并在服务器上使用 `srun --pty bash` 交互式申请 GPU 节点进行调试，确认 CUDA 可用后再提交批处理任务。

## 1. 华为云 ModelArts 部署踩坑

### 1.1 `copy_parallel` 目录嵌套陷阱

**日期**：2026-07-15

**现象**：同步 OBS 数据后，manifest 文件找不到：

```text
FileNotFoundError: '/cache/data/processed/train_manifest.json'
```

`find /cache -name "train_manifest.json"` 发现实际路径是：

```text
/cache/data/processed/processed/train_manifest.json
```

**原因**：`moxing` 的 `mox.file.copy_parallel(src, dst)` 会将源目录**本身**也复制到目标路径下，而不是仅复制内容。当执行：

```python
mox.file.copy_parallel(
    "obs://.../processed/",
    "/cache/data/processed/"
)
# 结果: /cache/data/processed/processed/  ← 双层嵌套
```

**解决**：目标路径设为 `/cache/data/`，让 OBS 上的 `processed/` 目录直接被放到该位置：

```python
mox.file.copy_parallel(
    "obs://.../processed/",
    "/cache/data/"
)
# 结果: /cache/data/processed/  ← 正确
```

**教训**：`moxing.copy_parallel` 的语义是"把 src **这个目录** 放到 dst 下"，不是 rsync 式的"把 src 的**内容**同步到 dst"。首次使用时先用 `ls` 确认实际路径结构。

## 3. Bash `python -c` 双引号语法错误

**日期**：2026-07-15

**现象**：终端中 `python -c "..."` 双引号内的 `$` 和括号被 bash 解析，导致语法错误：

```text
syntax error near unexpected token `('
```

**原因**：bash 在双引号字符串中仍会展开变量 `$VAR` 和命令替换 `$(cmd)`。当 Python 代码包含 `os.path.join(...)` 或 f-string `f"{var}"` 时，括号和 `$` 被 bash 错误解析。

**解决**：用**单引号**包裹 Python 代码块，内部字符串使用双引号：

```bash
# ❌ 双引号 — bash 解析 $ 和 ()
python -c "...${...}..."

# ✅ 单引号 — 原样传递给 Python
python -c '...f"{}"...'
```

## 4. Python f-string 不能含反斜杠转义

**日期**：2026-07-15

**现象**：

```text
SyntaxError: f-string expression part cannot include a backslash
```

**原因**：Python 3.x 的 f-string 花括号 `{}` 内不允许出现 `\"` 等反斜杠转义序列。这是 Python 语言设计的限制（PEP 498），因为 `\` 是字符串字面量的转义符，而 f-string 表达式内部是独立解析的。

**错误写法**：

```python
print(f"缺失: {record[\"signal_file\"]}")  # ❌ SyntaxError
print(f"{split}: {old} → {data[\"count\"]}")
```

**解决**：将字典索引取值提前赋值给一个变量，再在 f-string 中引用：

```python
sig = record["signal_file"]
print(f"缺失: {sig}")                       # ✅ OK

new_count = data["count"]
print(f"{split}: {old} → {new_count}")      # ✅ OK
```

### 1.5 NumPy 2.x 不兼容导致 NPU 初始化失败

**日期**：2026-07-15

**现象**：冒烟测试启动时报错，多重 import 链崩溃：

```text
AttributeError: _ARRAY_API not found
ImportError: numpy.core.multiarray failed to import
RuntimeError: Initialize: NPU function error, error code is 500001
EC0010: Failed to import Python module [AttributeError: `np.float_` was removed in the NumPy 2.0 release]
```

关键日志：`GEInitialize failed. ge result = 4294967295`，CANN 图编译引擎因 NumPy 版本不兼容完全无法启动。

**原因**：ModelArts 预置环境（PyTorch 2.1.0 + CANN 8.0.0 + torch_npu 2.1.0.post10）全部基于 NumPy 1.x 编译。NumPy 2.0 移除了 `np.float_` 等旧别名，所有 C++ 扩展（torch、wfdb、pandas、onnxruntime、torch_npu）在 import 时因找不到 NumPy C API 符号而崩溃。

项目的 `requirements.txt` 中 `numpy>=1.24.0` 允许 pip 安装 NumPy 2.2.6，导致覆盖了预置 NumPy。

**解决**：

```bash
pip install "numpy<2"
```

**教训**：在预置 AI 框架的云平台上，务必先检查预置 PyTorch 版本的 NumPy 兼容性。`pip install -r requirements.txt` 会静默升级 NumPy，需要将其放在 `numpy<2` 降级之后执行。

**正确顺序**：

```bash
pip install "numpy<2"      # 先降级
pip install -r requirements.txt  # 再装其他依赖
```

### 1.6 预处理数据含 NaN 导致训练全量静默失败

**日期**：2026-07-15

**现象**：冒烟测试跑通（无异常退出），但评估指标等于随机：

| 指标 | 实际值 | 随机基线 |
| ---- | ------ | -------- |
| val_auc | 0.5000 | 0.5000 |
| macro_f1 | 0.1169 | ~0.12 |
| mAP | 0.0708 | ~0.07 |

同时日志中出现：`logits包含 174474 NaN/Inf值，已替换为0`（验证集 6,462 × 27 = 174,474，即 100% 的 logits 都是 NaN）。

**排查过程**：

1. 怀疑 AMP 混合精度 → 加 `--no-amp`，无效
2. 怀疑 AsymmetricLoss 数值不稳定 → 切换 BCEWithLogitsLoss，无效
3. 怀疑 BatchNorm running stats 被污染 → 干净模型评估正常（macro_auc=0.5483），排除
4. 逐 batch 检查输入数据 → **Batch 89 输入信号 min=max=mean=NaN**

    ```python
    for batch_idx, (signals, labels) in enumerate(loader):
        if torch.isnan(model(signals.to(device))).any():
            # signal min=nan max=nan mean=nan has_nan=True
    ```

5. 全量扫描数据 → **8 / 43,101 个 .npy 文件**（0.019%）含 4,096 个全导联 NaN

**根因**：4 环因果链：

```text
8 个损坏 .npy 文件含 NaN（预处理异常未过滤）
  → Dataset.__getitem__ 无 NaN 清洗，直接进入训练
  → 部分 batch 的 NaN 被代码检测到并跳过 optimizer.step
    但 backward() 已执行，梯度中含 NaN，静默污染模型权重
  → 评估时污染后的权重在 FP32 下产 100% NaN logits
```

**解决**：在三个 Dataset 类的 `np.load()` 后统一添加 `np.nan_to_num()`：

```python
# 修复前
signal = np.load(signal_path).astype(np.float32)

# 修复后
signal = np.nan_to_num(np.load(signal_path).astype(np.float32),
                       nan=0.0, posinf=0.0, neginf=0.0)
```

涉及文件：[dataset.py](../src/data_pipeline/dataset.py) 的三个 `__getitem__` 方法。

**修复后结果**（2 epoch 冒烟测试）：

| 指标 | 修复前 | 修复后 |
| ---- | ------ | ------ |
| NaN batch | 3-4 个 | 0 |
| logits NaN | 174,474 | 0 |
| val_auc | 0.5000 | **0.5909** |
| macro_auc (test) | 0.5000 | **0.5879** |

**教训**：

- 训练 NaN 保护机制（跳过坏 batch）有根本缺陷 — `backward()` 之后才检测 NaN，梯度污染已经发生
- 数据清洗应在数据加载层（Dataset）完成，而非依赖训练层的检测
- 两轮 174K 全量 NaN 都未被发现异常，因为 NaN 保护代码把 logits 静默替换为 0，评估继续执行，metric 恰好等于随机基线，不报错
- 下次遇到"评估 = 随机基线"，首先排查输入数据质量

### 1.8 早停基于 val_f1 导致过度敏感 — InceptionTime 11 epoch 早停

**日期**：2026-07-15

**现象**：InceptionTime 50 epoch 训练在 epoch 11 触发早停，val_auc 仅到 0.61。

**原因**：早停的判断指标是 `val_f1`（阈值 0.5），但 27 类标签严重不平衡（485:1），BCE loss 的预测概率全部 <0.5，val_f1 全程卡在 0.1181（随机基线）。从 epoch 2 开始 val_f1 不再改善，连续 10 轮触发早停。而实际上 val_auc 持续在涨（0.55→0.61），说明模型在有效学习，只是 F1 指标被阈值卡住了。

**解决**：将早停判断指标从 `val_f1` 改为 `val_auc`。val_auc 从 0.50 开始逐步上升，能真实反映模型改进。

**教训**：在极端不平衡分类任务中，基于阈值的指标（F1）不适合作为早停判断标准。应使用阈值无关的指标（AUC、mAP）。

### 1.9 xResNet1D-101 学习率过高导致快速过拟合

**日期**：2026-07-15

**现象**：xResNet1D-101 用 `lr=1e-4` 训练：

```
Epoch 1: loss 0.083→0.002（在 235 batch 内卷完），val_auc=0.596
Epoch 2-7: loss 0.0000，val_auc 从 0.584 跌到 0.563（过拟合）
```

**根因**：InceptionTime 的 lr=1e-4 被直接复用到 xResNet1D-101，但 xResNet 参数量是 InceptionTime 的 30 倍（7.6M vs 291K），在 30K 样本上对 1e-4 过于敏感，1 个 epoch 就过拟合了训练集。

**解决**：lr 降为 1e-5。三种骨干的推荐学习率：

| 骨干 | 参数量 | 推荐 lr |
|------|--------|---------|
| InceptionTime | 252K | 1e-4 |
| xResNet1D-101 | 7.6M | 1e-5 |
| ECG Transformer | 6.5M | 1e-5 |

**教训**：学习率不应在不同参数规模的模型间直接复用。经验公式：参数量 ×10 → lr ÷10。

### 1.10 AsymmetricLoss 实现与论文公式不一致（代码审查发现）

**日期**：2026-07-15

**说明**：全量代码审查发现，ASL 的负类聚焦权重实现与论文（Ben-Baruch et al., ICCV 2021）不同：

- 论文：`(max(p-m, 0))^gamma_neg` — 给难负样本（高概率）更高权重
- 代码：`(1-p)^gamma_neg` — 给难负样本更低权重

当 gamma_neg=4，p=0.9（真负类），论文权重 0.52（高惩罚），代码权重 0.0001（几乎静默）。代码注释声称「正确实现了论文概率偏移」不准确。

**影响**：公式不一致但不崩溃。对极端不平衡任务的具体影响需消融实验验证。论文写作时应采用实际使用的公式。

**来源**：详见 [代码审查报告](代码审查报告.md) #4。

## 2. 训练调优踩坑

### 2.1 InceptionTime 过拟合 — 正则化治标不治本

**日期**：2026-07-15

**现象**：InceptionTime (291K 参数) 训练 3 轮 train loss 从 0.08 骤降到 0.003，val_auc 峰值仅 0.59 后开始下降。

**尝试**：
- weight_decay 1e-3 + dropout 0.3 + label_smoothing 0.05 + grad_noise 0.005 → val_auc 反降到 0.50（正则化过强压死学习）
- weight_decay 5e-4 + dropout 0.2 + label_smoothing 0.05 → val_auc 回到 0.59（正则化减轻但峰值不变）

**根因**：291K 参数是硬瓶颈。无论怎么调正则化，模型容量不足以同时学好 27 类不平衡分类。val_auc ~0.59 就是 InceptionTime 的数据集天花板。

**教训**：小模型过拟合时，正则化只能延缓无法突破上限。该换大模型时就换。

---

### 2.2 xResNet lr=1e-5 + ASL 导致崩溃式过拟合

**日期**：2026-07-15

**现象**：xResNet1D-101 (7.6M) 用 `lr=1e-5, AsymmetricLoss` 训练：

```
Epoch 1:  val_auc=0.540
Epoch 3:  val_auc=0.583  ← 峰值
Epoch 4+: val_auc 持续下跌到 0.544  ← 比 InceptionTime 还差
Test macro_auc: 0.544
```

**根因**：两个问题叠加：
1. **lr=1e-5 太低**：大模型用小 lr，快速收敛到尖锐极小值而非平坦极小值，泛化极差
2. **AsymmetricLoss + label_smoothing 冲突**：ASL 的 `(1-p)^gamma_neg` 概率偏移与标签平滑的软标签方向相反

**解决**：
- 换 BCEWithLogitsLoss（数值更稳定）
- lr 提高到 5e-5（避免尖锐极小值）
- 加强正则化（wd=1e-3, drop=0.3, ls=0.1）

**结果**：test macro_auc 从 0.544 飙升到 **0.791**。

**教训**：
- 大模型 lr 不应一味降低。经验公式"参数×10 → lr÷10"在此失效
- AsymmetricLoss 与标签平滑不兼容，选一个用
- 相同症状（val_auc 下跌）可能有完全不同根因——前一个是小模型容量不够，这一个是大模型优化器配置错误

---

### 2.3 梯度噪声公式 bug — 噪声被放大 1e5 倍

**日期**：2026-07-15

**现象**：第一次正则化实验时 val_auc 从 0.55 直接掉到 0.50（随机基线），train loss 反而正常下降。

**代码**：
```python
# ❌ bug: eta / lr = 1.0 / 0.0001 = 10000
eta = lr / (1 + self.current_epoch) ** 0.55
p.grad.add_(noise * eta / lr)  # 噪声被放大 10000 倍
```

**根因**：梯度噪声公式错误。`eta` 已经按学习率缩放，又除了一个 `lr`，导致噪声放大 4 个数量级。等价于向梯度注入 std=50 的噪声而非 std=0.005。

**修复**：
```python
# ✅ 正确: 噪声标准差随时间衰减
eta_t = 1.0 / (1 + self.current_epoch) ** 0.55
noise_std = grad_noise * eta_t
p.grad.add_(torch.randn_like(p.grad) * noise_std)
```

**教训**：新功能首次实验时先跑 `--quick-test`（2 epoch），对比有无该功能的差异。如果加了正则化 val_auc 反而归零，99% 是代码 bug 而非"正则化太强"。

### 2.4 早停从 val_f1 改为 val_auc — 但 min_delta 也需要

**日期**：2026-07-15

**现象**：xResNet 训练中 epoch 25 val_auc 已达 0.791，但早停在 epoch 46 才触发——中间 21 轮纯浪费。

**根因**：val_auc 在 0.791-0.793 之间微小波动，每次 +0.001 的"改善"都重置早停计数器。patience=10 但 min_delta=0，等同于"任何改善都算改善"。

**修复**：
```python
# 修复前: 微小波动就重置计数器
if val_metrics["macro_auc"] > best_val_auc:

# 修复后: 改善不足 0.0001 不重置
if val_metrics["macro_auc"] > best_val_auc + 1e-4:
```

**教训**：基于 AUC 的早停必须设 min_delta。AUC 在收敛后波动 ~0.001 是正常的，不设 min_delta 等于让早停失效。

### 2.5 数据增强在多进程 DataLoader 中产生相同序列（Windows）

**日期**：2026-07-15

**现象**：所有 DataLoader worker 的 ECGAugmentor 产生相同增强序列。

**根因**：`ECGAugmentor(random_seed=42)` 在 `train_backbone.py` 只创建一次，通过 pickle 传给所有 worker。`Generator` 实例的初始状态相同，`_worker_init_fn` 只设置 `np.random.seed()`，不影响独立的 Generator。

**影响**：同一样本在不同 worker 中的增强完全一致，数据多样性降低。Linux fork 模式下自然分叉，Windows spawn 模式受影响。

**状态**：📝 记录，Notebook (Linux) 不受影响，本地 Windows 训练时需注意。

## 3. 评估指标踩坑

### 3.1 F1 阈值 0.5 一刀切 — 严重低估模型能力

**日期**：2026-07-15

**现象**：test macro_auc=0.791，但 macro_f1 只有 0.139。

**根因**：27 类严重不平衡（485:1），BCE loss 输出的概率分布因类而异。常见类（窦性心律）概率集中在 0.7-0.9，稀有类（窦性心律失常 43 样本）概率集中在 0.1-0.3。阈值 0.5 一刀切 → 稀有类全部判阴性 → F1=0。

**修复**：逐类搜索最优阈值（P-R curve 上 F1 极大点）：
```python
prec, rec, thresh = precision_recall_curve(y_true, y_prob)
f1_curve = 2 * prec[:-1] * rec[:-1] / (prec[:-1] + rec[:-1] + 1e-8)
best_thresh = thresh[np.argmax(f1_curve)]
```

**结果**：macro_f1 从 0.139 提升到 0.291（翻倍）。

**教训**：不平衡分类任务中，AUC 是唯一不依赖阈值的可信指标。F1 必须配合逐类阈值调优才有意义。
